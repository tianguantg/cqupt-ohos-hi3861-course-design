# 系统设计方案

## 一、总体设计

### 1.1 设计目标

本系统面向家庭室内环境安全监测场景，基于 OpenHarmony 和 Hi3861 开发板实现温湿度、烟雾或可燃气体数据采集，并通过 OLED 屏幕进行本地显示。当环境数据超过设定阈值时，系统进入报警状态；在网络可用时，设备通过 MQTT 将实时数据上传至物联网云平台，便于远程查看。

### 1.2 系统架构

```text
[SHT30 温湿度传感器] --I2C-->
                                 [Hi3861 开发板] --Wi-Fi/MQTT--> [物联网云平台] --> [网页/手机端查看]
[烟雾/可燃气体传感器] --ADC-->      |
                                  +--SPI/GPIO--> [OLED 显示屏]
                                  +--GPIO/PWM--> [蜂鸣器或报警输出]
```

### 1.3 功能模块划分

| 模块编号 | 模块名称 | 功能说明 | 涉及硬件/协议 |
|---|---|---|---|
| M1 | 数据采集模块 | 采集温湿度、烟雾或可燃气体数据 | SHT30、气体/烟雾传感器、I2C、ADC |
| M2 | 本地显示模块 | 显示温度、湿度、气体/烟雾值和报警状态 | OLED、SPI/GPIO |
| M3 | 报警判断模块 | 根据阈值判断环境是否异常 | GPIO、蜂鸣器/LED |
| M4 | 网络通信模块 | 连接 Wi-Fi 并通过 MQTT 上传数据 | Wi-Fi、MQTT |
| M5 | 云端展示模块 | 展示设备状态、传感器数据和报警信息 | OneNET/腾讯云/阿里云 |

## 二、硬件设计

### 2.1 硬件清单

| 序号 | 名称 | 型号/规格 | 数量 | 用途 |

### 2.2 硬件连接

| 外设 | 通信方式 | 功能 |
|---|---|---|
| SHT30 | I2C | 温湿度数据读取 |
| OLED | SPI/GPIO | 字符和状态显示 |
| 烟雾传感器 | ADC/GPIO | 烟雾浓度或报警状态读取 |
| 可燃气体传感器 | ADC | 气体浓度采集 |
| 蜂鸣器/LED | GPIO/PWM | 报警提示 |
| Wi-Fi | Hi3861 内置 | 连接云平台 |

## 三、软件设计

### 3.1 软件架构

```text
应用层：环境监测主任务、报警逻辑、显示刷新、数据上报
服务层：MQTT 客户端、Wi-Fi 连接管理、数据格式封装
驱动适配层：OLED、SHT30、ADC、GPIO
硬件接口层：OpenHarmony Hi3861 HAL 接口
```

### 3.2 主程序流程

```c
int main(void)
{
    Board_Init();
    OLED_Init();
    SHT30_Init();
    GasSensor_Init();
    Alarm_Init();

    WiFi_Connect();
    MQTT_Connect();

    while (1) {
        Read_Temperature_Humidity();
        Read_Gas_Or_Smoke_Value();
        Check_Alarm_Status();
        Update_OLED_Display();
        Publish_Data_If_Connected();
        DelayMs(1000);
    }
}
```

### 3.3 数据结构设计

```c
typedef struct {
    float temperature;
    float humidity;
    int gasValue;
    int smokeValue;
    int alarm;
} EnvData;
```

### 3.4 阈值判断

| 指标 | 默认阈值 | 报警条件 |
|---|---|---|
| 温度 | 35 摄氏度 | temperature > 35 |
| 湿度 | 80%RH | humidity > 80 |
| 气体值 | 根据实测标定 | gasValue > GAS_THRESHOLD |
| 烟雾值 | 根据实测标定 | smokeValue > SMOKE_THRESHOLD |

阈值需要根据传感器实测数据调整，答辩时可以说明通过多次测试确定合适范围。

## 四、云平台设计

### 4.1 平台选择

建议优先选择 OneNET 或腾讯云 IoT Explorer。课程设计阶段可先完成 MQTT 上报，再考虑远程控制指令。

### 4.2 MQTT 数据格式

```json
{
  "temperature": 28.3,
  "humidity": 65.0,
  "gas": 380,
  "smoke": 120,
  "alarm": 0
}
```

### 4.3 上传数据项

| 数据项 | 类型 | 上传周期 | 说明 |
|---|---|---|---|
| temperature | float | 5s | 温度 |
| humidity | float | 5s | 湿度 |
| gas | int | 5s | 可燃气体检测值 |
| smoke | int | 5s | 烟雾检测值 |
| alarm | int | 5s | 0 表示正常，1 表示报警 |

## 五、开发计划

| 阶段 | 任务 | 输出 |
|---|---|---|
| 第 1 阶段 | 环境搭建、HelloWorld 编译烧录 | 串口输出正常 |
| 第 2 阶段 | OLED 示例调试 | OLED 能显示固定文本 |

| 第 5 阶段 | 阈值报警和系统集成 | 完成本地演示版本 |
| 第 6 阶段 | MQTT 云平台接入 | 云端查看实时数据 |
| 第 7 阶段 | 测试、文档和答辩准备 | 测试报告、截图、PPT |

## 六、风险评估

| 风险 | 影响 | 应对措施 |
|---|---|---|
| OLED 或传感器驱动调试失败 | 本地显示或采集不可用 | 先运行官方示例，逐个模块验证 |
| ADC 阈值不稳定 | 报警误触发 | 根据实测数据设置阈值并增加简单滤波 |
| Wi-Fi 或 MQTT 配置失败 | 云端功能不可演示 | 保留本地 OLED + 串口日志作为离线演示方案 |
| 时间不足 | 功能完成度下降 | 优先完成温湿度、OLED 和报警，云平台作为后续加分 |
